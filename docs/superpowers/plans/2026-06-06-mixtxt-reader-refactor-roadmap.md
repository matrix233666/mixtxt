# MixTXT Reader 重构路线图

## 背景

Reader Experience 子项目已经达到可交付状态，当前已具备以下能力：

- 顶层公共导航、`/books/`、`/about/`
- 更完整的书籍详情页与章节阅读页
- 阅读偏好持久化与 continue-reading
- 章节目录抽屉与移动端阅读布局
- 构建、类型检查和基础回归验证

接下来的重点不再是补功能，而是为下一阶段开发降低维护成本和回归风险。当前仓库最值得继续打磨的方向主要有 3 类：

1. 测试保护仍有一部分停留在源码字符串断言层
2. 客户端脚本逻辑分散在多个页面和组件中
3. 样式文件职责开始变宽，继续增长后会影响维护效率

本路线图只覆盖非阻塞重构，不引入新的产品需求，也不改变项目的 static-first 边界。

## 目标

本轮重构的目标是：

- 用更稳定的行为测试替代一部分脆弱的源码守卫
- 收口继续阅读与阅读器相关的客户端脚本
- 拆清样式文件职责，避免后续页面继续堆进 `global.css`
- 在不改变当前用户可见行为的前提下，为下一份 spec 做准备

## 非目标

本路线图不处理以下事项：

- 新增 reader 功能
- 搜索体验重设计
- 新的内容模型扩展
- 引入客户端框架
- 改变当前构建链路和静态站点模型

## 现状判断

当前实现的问题不在“能不能用”，而在“后续继续演进时会不会越来越难改”。

具体表现为：

- 部分回归测试依赖源码结构，重命名或改写实现方式时容易误报
- 首页、书籍页、阅读器工具栏各自带有一段浏览器端装配逻辑
- `/books/` 页筛选与排序逻辑仍然偏集中
- `global.css` 和 `reader.css` 的职责边界已经开始变得不够清晰

这些问题现在不会阻塞交付，但如果下一阶段继续扩展 reader 或书籍页交互，维护成本会明显上升。

## 重构原则

- 先补测试，再做结构调整
- 先收口重复逻辑，再拆样式职责
- 优先做低风险、可验证的小步重构
- 每个阶段结束后都必须保留当前页面行为不变
- 不为了“结构更漂亮”牺牲现有稳定性

## 分阶段任务

### 阶段一：把关键回归保护从源码断言升级为行为测试

#### 阶段一目标

让 reader 和 `/books/` 页的关键交互更多地由行为测试保护，而不是依赖源文件里必须出现某段字符串。

#### 阶段一主要改动范围

- `tests/build/reader-regressions.test.ts`
- `tests/build/routes.test.ts`
- 与 reader、continue-reading、`/books/` 筛选相关的构建或浏览器测试

#### 阶段一任务清单

- [ ] 盘点现有回归测试，区分哪些属于“行为测试候选”，哪些仍适合保留为源码守卫
- [ ] 为章节目录补行为测试，覆盖桌面端目录可见、移动端目录可打开且不遮挡正文
- [ ] 为 continue-reading 补行为测试，覆盖有效进度、失效进度、非公开章节进度
- [ ] 为 `/books/` 页补筛选和排序行为测试，覆盖状态筛选、标签筛选和排序生效
- [ ] 收缩 `reader-regressions.test.ts`，只保留少量确实不适合高层验证的守卫

#### 建议保留为源码守卫的内容

- 构建锁恢复阈值这类非常实现细节的保护
- 极小型内联 bootstrap 钩子，且难以从构建产物中稳定观测的场景

#### 阶段一验收标准

- reader 关键交互主要由行为测试覆盖
- stale `localStorage` 场景由用户可见行为验证保护
- `/books/` 页筛选与排序有明确的交互级测试
- `reader-regressions.test.ts` 不再承担过多实现细节校验

### 阶段二：收口阅读器与继续阅读相关的客户端脚本

#### 阶段二目标

减少页面内重复的 DOM 查询、嵌入 JSON 解析和 `localStorage` 装配逻辑，让页面文件更接近“组装页面”，而不是“同时处理视图和客户端状态”。

#### 阶段二主要改动范围

- `src/pages/index.astro`
- `src/pages/books/[book].astro`
- `src/pages/books/index.astro`
- `src/components/ReaderToolbar.astro`
- `src/layouts/BaseLayout.astro`
- `src/lib/reading.ts`

#### 阶段二任务清单

- [ ] 新增一个轻量客户端 helper 模块，例如 `src/lib/reader-client.ts`
- [ ] 把 continue-reading 目标解析逻辑统一放进客户端 helper
- [ ] 把阅读偏好应用到 DOM 的逻辑统一收口，避免 BaseLayout 和 Toolbar 各自维护一套
- [ ] 统一页面嵌入 JSON 数据的写法、读取方式和异常兜底
- [ ] 将 `/books/` 页脚本拆成“数据筛选排序”和“DOM 更新”两层
- [ ] 简化页面内联脚本，只保留初始化入口和页面级数据注入

#### 阶段二 helper 建议职责

适合收进客户端 helper 的逻辑包括：

- 读取嵌入页面的 JSON 数据
- 从本地进度解析 continue-reading 目标
- 将 reader 偏好应用到 `documentElement` 或 reader root
- 根据视口切换目录开合状态
- 为 `/books/` 页提供筛选和排序计算结果

#### 阶段二验收标准

- 首页和书籍页不再重复实现 continue-reading 装配逻辑
- ReaderToolbar 与 BaseLayout 使用同一套偏好应用规则
- `/books/` 页脚本职责更清晰，后续新增筛选项时不需要继续堆逻辑
- `.astro` 页面脚本明显变短，页面职责更明确

### 阶段三：按页面域和交互域拆分样式职责

#### 阶段三目标

在不改变当前视觉效果的前提下，让样式文件的职责边界更清楚，避免后续所有页面样式都继续堆到 `global.css` 或 `reader.css` 里。

#### 阶段三主要改动范围

- `src/styles/global.css`
- `src/styles/reader.css`
- `src/styles/search.css`
- 未来可新增的页面样式文件，例如 `books.css`、`about.css`

#### 阶段三任务清单

- [ ] 先定义样式职责边界，明确哪些规则属于全站壳层，哪些属于页面域
- [ ] 把 `global.css` 收缩到全站基础元素、通用布局、导航、页脚、通用按钮和通用 surface
- [ ] 把书籍页、关于页、书单页等样式按页面域拆出去
- [ ] 如果下一阶段 reader 继续增长，再把 `reader.css` 进一步拆成 layout、toolbar、directory、mobile override
- [ ] 每次样式迁移后都做构建输出和目标页面检查，防止出现静默回归

#### 阶段三建议样式边界

`global.css` 适合保留：

- reset 与基础元素
- 全站排版基线
- page shell、header、footer
- 通用按钮、badge、tag、surface

页面域样式适合拆分为：

- `books.css`
- `about.css`
- `reader.css`
- `search.css`

如果 reader 再扩张，可以继续细分为：

- `reader-layout.css`
- `reader-toolbar.css`
- `reader-directory.css`

#### 阶段三验收标准

- `global.css` 不再混入多个页面的大段专属样式
- reader 样式结构清楚，后续加交互时能快速定位
- 页面样式迁移后无明显视觉回归

## 推荐执行顺序

建议严格按下面顺序推进：

1. 先做测试形态升级
2. 再做客户端脚本收口
3. 最后做样式职责拆分

原因是：

- 没有更稳的测试，后续重构缺少保护
- 页面脚本不先收口，样式拆分容易和交互调整搅在一起
- 样式职责拆分最适合放在结构相对稳定之后做

## 风险与应对

### 风险一：测试重写期间覆盖断层

如果直接删除源码守卫，但行为测试还没补上，短期内可能出现保护空窗。

应对方式：

- 先补行为测试，再删源码守卫
- 每完成一类行为测试，再移除对应的源码字符串断言

### 风险二：脚本收口时引入页面行为差异

首页、书页和章节页的运行环境并不完全一样，统一 helper 时可能把页面差异处理掉，导致某一页行为退化。

应对方式：

- 先提取纯函数和通用 helper
- 页面初始化入口仍保留在各自页面中
- 通过针对页面的行为测试验证回归

### 风险三：样式拆分导致静默视觉回归

CSS 文件移动后，容易出现选择器优先级变化或加载顺序变化。

应对方式：

- 每次只迁移一个页面域或一个 reader 子域
- 迁移后立即跑 build，并检查关键页面
- 对移动端 reader、桌面端目录、书单页筛选结果做定向检查

## 暂停条件

出现以下任一情况时，应暂停这份路线图，先回到功能稳定：

- 新的产品 spec 需要优先开发新功能
- 行为测试补充过程中暴露出当前实现里的未修复问题
- 样式拆分已经开始影响当前页面视觉输出

暂停后应优先解决功能或回归问题，再继续推进结构优化。

## 完成信号

这份路线图可以视为完成，当且仅当：

- reader 与 `/books/` 页的关键回归主要由行为测试保护
- continue-reading 与 reader 偏好相关客户端逻辑已经统一收口
- 页面内联脚本明显缩短，只负责页面初始化
- `global.css`、`reader.css` 与页面域样式职责边界清楚
- 在完成每个阶段后，构建、检查与测试都保持通过

## 每阶段验证清单

每完成一个阶段，至少执行以下验证：

- `npm test`
- `npm run check`
- `npm run build`

如果阶段内容涉及交互或移动端布局，还应补充定向检查：

- 章节目录在桌面端和移动端的行为是否符合预期
- continue-reading 在有效与失效进度下是否表现正确
- `/books/` 页筛选和排序是否仍然正确
- 移动端 reader 的间距和正文可读性是否保持稳定

## 交付建议

如果只打算做一项非阻塞优化，优先做：

- 阶段一：行为测试升级

如果准备做两项，建议组合为：

- 阶段一：行为测试升级
- 阶段二：客户端脚本收口

如果要为下一个 reader 子项目提前清理地基，再继续做：

- 阶段三：样式职责拆分
